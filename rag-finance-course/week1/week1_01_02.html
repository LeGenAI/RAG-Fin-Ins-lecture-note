RAG 시스템 아키텍처: 5단계 처리 파이프라인

Indexing Phase (데이터 준비 단계)

1. 문서 불러오기: PDF, 웹페이지, 데이터베이스에서 자료 가져오기
2. 텍스트 나누기: 긴 문서를 작은 조각으로 분할 (약 500-1000 단어)
3. 벡터로 변환: 텍스트를 숫자(벡터)로 변환해서 컴퓨터가 이해하도록
4. 벡터 저장: 변환된 벡터를 데이터베이스에 저장

핵심 고려사항

• 조각 크기: 너무 작으면 전체 내용을 못 담고, 너무 크면 정확한 검색이 어려워짐

• 겹치기 설정: 조각 사이를 약간 겹쳐서 중요한 정보가 끊어지지 않도록

• 추가 정보: 문서 출처, 작성 날짜 등을 함께 저장

처리 성능 벤치마크

100만 문서 인덱싱: 2시간 | 벡터 검색 속도: <50ms | 임베딩 차원: 768-1536

텍스트 분할: 큰 문서를 작은 조각으로 나누기

왜 문서를 나눠야 할까요?

1. 정확성: "구글이 앤스로픽에 투자한 금액은?" 같은 질문에 전체 100페이지 문서가 아닌 해당 정보만 있는 1단락만 찾아서 답변

2. 비용 절감: GPT-4에 100페이지 전체를 보내면 약 $1, 필요한 2-3단락만 보내면 $0.01

3. 할루시네이션 방지: 너무 많은 정보를 주면 LLM이 헷갈려서 엉뚱한 답변을 생성할 수 있음

Chunk Size와 Overlap 이해하기

예시 문서:
"서강대학교는 1960년에 설립된 대학입니다. | 수학과는 1964년에 개설되었으며, 현재 20명의 교수진이 있습니다. | 김종락 교수는 인공지능과 수학의 융합 연구를 진행하고 있습니다."

Chunk Size = 50자: 각 조각의 최대 길이
Chunk Overlap = 10자: 조각 간 겹치는 부분

조각 1: "서강대학교는 1960년에 설립된 대학입니다. 수학과는 1964년에"
조각 2: "수학과는 1964년에" "개설되었으며, 현재 20명의 교수진이"
조각 3: "20명의 교수진이" "있습니다. 김종락 교수는..."

실무 권장 설정

일반 문서: chunk_size=1000, overlap=200

금융 약관: chunk_size=800, overlap=100

뉴스 기사: chunk_size=500, overlap=50

코드로 이해하는 문서 청킹

from langchain_text_splitters import RecursiveCharacterTextSplitter

# 단계 2: 문서 분할
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,      # 각 조각의 최대 크기
    chunk_overlap=50,     # 조각 간 겹치는 부분
    separators=[          # 우선순위별 분할 기준
        "\n\n",          # 1순위: 단락
        "\n",            # 2순위: 줄바꿈
        ".",             # 3순위: 문장
        " ",             # 4순위: 단어
    ]
)

# 문서를 조각으로 분할
splits = text_splitter.split_documents(docs)
print(f"총 {len(splits)}개의 조각으로 분할됨")

임베딩: 텍스트를 숫자(벡터)로 변환하는 마법

임베딩이란 무엇인가?

임베딩이 왜 필요한가요?

의미 이해: 컴퓨터는 "주택담보대출"이라는 텍스트를 이해할 수 없지만, [0.1, 0.5, 0.9, ...]같은 숫자는 계산할 수 있습니다

유사도 계산: "주택담보대출"과 "주담대"가 비슷한 의미임을 숫자의 거리로 파악할 수 있습니다

쉬운 비유:
임베딩은 단어를 지도 위의 좌표로 바꾸는 것과 같습니다.
• "은행"과 "금융기관" → 가까운 위치
• "은행"과 "의자" → 먼 위치
• "강가"와 "둑" → 중간 거리 (다른 의미의 은행)

임베딩 실제 예시: 유사도 계산 과정

질문: "시장조사기관 IDC가 예측한 AI 소프트웨어 시장의 연평균 성장률은?"

🎯 문서 1 (최고 유사도)

"IDC는 AI 시장이 연평균 31% 성장할 것으로 예측..."

벡터: [0.1, 0.5, 0.9, ..., 0.1, 0.2]

유사도: 0.85 (85%)

문서 2

"국내 은행들의 디지털 전환 현황..."

벡터: [0.7, 0.1, 0.3, ..., 0.5, 0.6]

유사도: 0.30 (30%)

문서 3

"블록체인 기술의 금융 적용 사례..."

벡터: [0.9, 0.4, 0.5, ..., 0.4, 0.3]

유사도: 0.25 (25%)

결과: 문서 1이 85% 유사도로 가장 높음 → RAG 시스템이 이 문서를 컨텍스트로 사용하여 답변 생성

코드로 이해하는 임베딩

from langchain_openai import OpenAIEmbeddings
import numpy as np

# 임베딩 모델 초기화
embeddings = OpenAIEmbeddings()

# 텍스트를 벡터로 변환
question = "IDC가 예측한 AI 시장 성장률"
question_vector = embeddings.embed_query(question)
print(f"벡터 차원: {len(question_vector)}")  # 1536

# 문서들도 벡터로 변환
doc1 = "IDC는 AI 시장이 연평균 31% 성장..."
doc1_vector = embeddings.embed_query(doc1)

doc2 = "국내 은행들의 디지털 전환 현황..."
doc2_vector = embeddings.embed_query(doc2)

# 코사인 유사도 계산
def cosine_similarity(v1, v2):
    return np.dot(v1, v2) / (
        np.linalg.norm(v1) * np.linalg.norm(v2)
    )

sim1 = cosine_similarity(question_vector, doc1_vector)
sim2 = cosine_similarity(question_vector, doc2_vector)

print(f"문서1과의 유사도: {sim1:.2f}")  # 0.85
print(f"문서2와의 유사도: {sim2:.2f}")  # 0.30

# 가장 유사한 문서 선택
if sim1 > 0.7:
    print("✓ 관련 문서 발견!")
    print(f"답변: {doc1}")

왜 RAG가 필요한가? - 핵심 가치

  • ✅ 정확성: 환각(Hallucination) 현상을 40% 감소시킴
  • ✅ 신뢰성: 모든 답변에 출처 문서 제공
  • ✅ 최신성: 실시간 문서 업데이트 가능
  • ✅ 경제성: LLM 재학습 없이 지식 확장

• 검색 시간: 90% 단축

• 정확도: 25% 향상

• 비용: 70% 절감

실제 동작 예시: 금융 상담

질문: "주택담보대출 금리가 어떻게 되나요?"

RAG 처리 과정

1️⃣ 질문을 벡터로 변환 (임베딩)
2️⃣ 관련 금융 문서 3개 검색 (유사도 0.85+)
3️⃣ 검색된 문서를 컨텍스트로 제공
4️⃣ LLM이 컨텍스트 기반으로 답변 생성

AI 답변:

"현재 주택담보대출 금리는 변동금리 연 3.5~5.2%, 고정금리 연 4.0~5.5%입니다."
